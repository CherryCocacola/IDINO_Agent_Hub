## Critical Issues

### Issue 1: `POST /reports/generate` returns HTTP 500 when called with a Jinja2 template_id
- **Severity:** HIGH
- **Steps to Reproduce:**
  1. `POST /api/v1/auth/login` with admin credentials → get token
  2. `GET /api/v1/templates` → note any template ID (e.g., `16c7ea35-807a-4955-8192-d4a839c7e6d7`)
  3. `POST /api/v1/reports/generate` with body `{"title":"Test","template_id":"16c7ea35-...","source_document_ids":[...],"output_format":"docx"}`
  4. **Result: HTTP 500 Internal Server Error**
- **Root Cause:** `GET /api/v1/templates` returns records from the `DocumentTemplate` (Jinja2) table, but `POST /reports/generate` looks up `template_id` in the `ReportTemplate` table. These are separate tables. When a Jinja2 template UUID is passed, no `ReportTemplate` row is found, and unhandled None attribute access causes a 500.
- **Note:** Free-form generation (`template_id: null`) works correctly and produces a valid DOCX.
- **Note:** There are currently **0 records** in `/reports/templates` (legacy ReportTemplate), so this bug is always triggered when a template is selected via UI.
- **Suggested Fix:**
  1. Add explicit guard in router or service: if `template_id` provided and no `ReportTemplate` found → raise `HTTPException(404, "보고서 템플릿을 찾을 수 없습니다")`.
  2. Investigate whether the Jinja2 `DocumentTemplate` should be the single template system — if so, update `reports/generate` to look up `DocumentTemplate` instead of `ReportTemplate`.
