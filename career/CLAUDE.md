# Session Convention Protocol

세션 시작 시 자동으로 로드되는 프로젝트 규약.

---

## 1. 언어 규약

- 사용자가 한국어로 말하면 한국어로 응답한다.
- 플랜 설명, 작업 결과 보고, 에이전트 요약 등 모든 커뮤니케이션은 **한글**로 진행한다.
- 코드 주석은 **초보자도 쉽게 이해할 수 있는 한글**로 반드시 작성한다.
  - 함수/클래스 위에 역할과 목적 설명
  - 복잡한 로직에 단계별 한글 설명
  - "왜(why)" 이렇게 하는지를 중심으로 작성
- 변수명, 함수명, 커밋 메시지는 영어를 사용한다.
- 기술 용어는 원어(영어) 그대로 사용한다 (한글 음차 허용: 컴포넌트, 마이크로서비스 등).

---

## 2. 에이전트 위임 규칙

아래 8개 전문 에이전트가 등록되어 있다. **해당 도메인의 작업이면 반드시 Task 도구로 위임**하고, 직접 처리하지 않는다.

| 에이전트 | 위임 조건 | 모델 |
|---|---|---|
| **backend-specialist** | API 설계, 서버 아키텍처, 인증/인가, 미들웨어, 캐싱, 큐, 서버사이드 디버깅 | opus |
| **frontend-specialist** | UI/UX 구현, 컴포넌트, 스타일링, 접근성, 브라우저 호환성, 프론트엔드 성능 | opus |
| **database-architect** | DB 스키마 설계, 쿼리 최적화, 마이그레이션, 인덱싱, 데이터 모델링 | opus |
| **data-analyst** | 데이터 분석, 통계, 데이터 정제, 시각화, pandas/SQL 작업 | opus |
| **research-assistant** | 기술 조사, 대안 비교, 트레이드오프 분석, 리서치 보고서 | sonnet |
| **sdet-agent** | 테스트 작성/리뷰/디버깅, 테스트 전략, CI/CD 테스트, 코드 작성 후 자동 호출 | opus |
| **enterprise-architect** | 시스템 아키텍처, 설계 패턴, 기술 스택 결정, 확장성 설계, 서비스 분해 | opus |
| **code-analysis-specialist** | 코드 품질 분석, 버그 탐지, 보안 취약점, 성능 이슈, 기존 코드 이해 (코드 작성 X, 분석 전용) | opus |

### 위임 원칙

- 작업이 단일 도메인에 속하면 → 해당 에이전트 1개에 위임
- 작업이 복수 도메인에 걸치면 → 관련 에이전트들을 **병렬로** 동시 실행
- 코드를 작성/수정한 후에는 → **sdet-agent를 자동으로** 호출하여 테스트 커버리지 확보
- 기존 코드를 분석/리뷰할 때 → **code-analysis-specialist**에 위임 (코드 작성은 이 에이전트가 하지 않음)
- 리팩토링 전 → code-analysis-specialist로 현황 분석 먼저, 그다음 구현 에이전트
- 아키텍처 결정이 필요한 큰 작업 → enterprise-architect 먼저, 그다음 구현 에이전트
- 에이전트 실행 결과는 사용자에게 **한글로 요약하여 보고**한다

---

## 3. 작업 수행 규약

### 코드 작성
- 기존 코드를 먼저 읽고 패턴을 파악한 뒤 수정한다
- 프로젝트의 기존 컨벤션(네이밍, 구조, 라이브러리)을 따른다
- **모든 코드에 초보자도 이해할 수 있는 한글 주석을 반드시 작성한다**
- 요청받지 않은 리팩토링, 기능 확장을 하지 않는다
- 보안 취약점(XSS, SQL injection, 인젝션 등)을 절대 도입하지 않는다

### 파일 관리
- 새 파일 생성은 최소화하고, 기존 파일 수정을 우선한다
- README, 문서 파일은 명시적 요청 없이 생성하지 않는다

### Git 운영
- 커밋은 사용자가 명시적으로 요청할 때만 수행한다
- push, force push, reset --hard 등 위험 명령은 반드시 확인 후 실행한다
- git config 을 변경하지 않는다

### 계획 수립
- 3개 파일 이상 수정이 필요한 작업 → EnterPlanMode로 **한글 계획서** 제시 후 승인받고 진행
- 단순 수정(1~2파일) → 바로 실행
- 계획 설명 시 각 단계의 목적과 이유를 한글로 명확히 서술한다

---

## 4. 응답 스타일

- 간결하고 핵심만 전달한다
- 이모지를 사용하지 않는다 (사용자 요청 시에만)
- 코드 참조 시 `파일경로:라인번호` 형식을 사용한다
- 시간 예측이나 추정을 하지 않는다
- 불확실한 정보를 추측하지 않고, research-assistant에 위임하거나 사용자에게 확인한다
- **결과 보고, 오류 설명, 선택지 제안 등 모든 출력은 한글로 작성한다**
